# 02-projects — Active Projects

**Purpose:** One folder per active project. Each folder is self-contained: overview, clients, processes, and a link to the Dev Brain code index.

## What goes here

- Any project with recurring work — client sites, personal tools, open-source repos, long-term initiatives
- Each project gets its own subfolder: `02-projects/<project-slug>/`

## What doesn't go here

- Archived or completed projects → move to `09-archive/<project-slug>/`
- One-off tasks — those live in daily notes or inbox captures
- Reference material about a project's domain — that goes in `03-knowledge/` or `04-resources/`

## Project folder structure

```
02-projects/<project-slug>/
├── _overview.md          ← single source of truth (status, strategy, decisions)
├── clients/              ← one .md per client (for service/agency projects)
│   └── <client-slug>.md
├── processes/            ← SOPs, runbooks, repeatable workflows
│   └── deploy-sop.md
└── graph-report.md       ← auto-generated by Dev Brain post-commit hook (do not edit)
```

## `_overview.md` is the source of truth

The `sync-overviews` skill and `project-context` skill both read `_overview.md`. Keep it current. It should answer: *What is this? Where does it stand? What's blocked? What's next?*

See `02-projects/_TEMPLATE/_overview.md` for the standard template.

## Naming

- Project slugs: `kebab-case` matching the git repo name where possible
- Client files: `<client-slug>.md` inside `clients/`
- Processes: descriptive kebab-case (`deploy-sop.md`, `client-onboarding.md`)

## Connecting to Dev Brain

When a coding project has a `.brain/` folder and graphify is installed, the Dev Brain pipeline auto-indexes that project. The `project-context` skill reads `~/Documents/Dev Brain/sessions/index.md` to pull recent coding sessions into the project brief.

See `03-knowledge/permanent-notes/per-project-second-brain.md` for the full per-project architecture.
